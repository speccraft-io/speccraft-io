---
title: The problems
description: The correctness problems durable-workflow developers actually hit — logical races, slow activities, initialization races, dangling handlers, invalid state combinations, and two-store drift.
---

Durable-execution engines already handle the failures that scared you into using them — crashes, retries, lost messages, non-deterministic replay. What they don't handle is your workflow's own logic going wrong under an ordering you never pictured. That class:

- **Logical races.** Domain invariants that quietly break under an interleaving nobody exercised — a signal that arrives one step earlier than assumed, two handlers that both read-then-write the same field. They live in the orderings no test suite samples, so they pass every test and fail in production.
- **Slow or surprising activities.** An activity takes longer than expected, retries, or completes out of the order you had in mind — and the workflow lands in a state you never designed for. A timer fires mid-handler; a "brief" call outlives the decision that depended on it finishing first.
- **Initialization races.** A signal arrives before setup finishes and mutates state that isn't there yet.
- **Dangling handlers.** Fire-and-forget work and unwaited signal handlers outlive the run — the workflow completes while a handler is still mid-write.
- **Invalid state combinations.** Independent flags — `submitted`, `deleted`, `converted` — drift into a combination the design never intended and no branch handles.
- **Two-store drift.** The workflow's state and a *second* stateful entity — an external store, a peer or child workflow — fall out of sync under interleaving. The hardest class, because correctness now depends on something the workflow doesn't control.

The first five are *intra-workflow*: they depend only on a single workflow's own bookkeeping, and they're largely tractable. The last is *cross-process*, and it's the unsolved core.
